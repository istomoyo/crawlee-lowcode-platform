# 毕业设计答辩准备笔记

## 1. 项目名称

基于 Crawlee 的低代码爬虫平台设计与实现

## 2. 项目定位

本项目面向有数据采集需求、但不希望每次都手写爬虫脚本的用户，设计并实现了一套低代码爬虫平台。  
用户可以通过可视化方式完成任务创建、页面结构预览、字段映射、执行控制、结果查看、模板复用和任务管理，从而降低网页数据采集的开发门槛。

如果用一句话概括本项目：

> 这是一个“让网页采集从写代码变成配配置”的低代码爬虫系统。

---

## 3. 选题背景与意义

### 3.1 背景

传统网页爬虫开发通常需要开发者手动分析页面结构、编写选择器、处理翻页逻辑、维护执行流程，开发成本较高，重复劳动较多。  
而在实际应用中，很多采集任务的逻辑具有较强相似性，例如：

- 打开目标网页
- 预览页面结构
- 选择列表节点
- 提取文本、图片、链接字段
- 执行任务并查看结果

因此，有必要设计一套低代码平台，将这些重复性的采集流程抽象为通用配置流程，提高开发效率与复用能力。

### 3.2 意义

- 降低爬虫任务开发门槛
- 提高采集任务的复用性和管理性
- 为后续扩展模板中心、通知系统、任务组织等能力提供基础
- 体现低代码思想在数据采集场景中的实际应用价值

---

## 4. 项目目标

本项目的主要目标包括：

1. 设计并实现一个可运行的低代码爬虫平台
2. 支持通过可视化方式创建网页采集任务
3. 支持任务执行、结果查看、模板复用与后台管理
4. 支持带登录态页面的采集能力
5. 在本科毕业设计范围内，完成从前端界面到后端接口的完整闭环

## 6. 系统总体架构

本系统采用前后端分离架构。

### 6.1 前端

前端基于 Vue 3 + TypeScript + Vite 构建，负责：

- 页面展示
- 表单交互
- 可视化任务配置
- 任务状态呈现
- 模板、通知、工作台等业务页面展示

### 6.2 后端

后端基于 NestJS 构建，负责：

- 用户认证与权限校验
- 任务配置接收与处理
- 爬虫任务调度与执行
- 结果文件管理
- 模板、通知、后台管理等接口提供

### 6.3 执行引擎

系统核心采集能力基于 Crawlee 与 Playwright 实现，负责：

- 页面访问
- 截图预览
- DOM 解析
- 字段提取
- 带 Cookie 的登录态访问
- 采集流程执行

### 6.4 数据存储

- MySQL：存储用户、任务、执行记录、模板、通知等结构化数据
- Redis：存储验证码、登录状态等临时数据
- 本地文件系统：存储截图、结果文件、打包文件等资源

---

## 7. 核心业务流程

### 7.1 任务创建流程

1. 用户输入任务名称与目标 URL
2. 系统对页面进行截图预览
3. 用户选择列表区域与字段节点
4. 用户配置抓取字段、翻页或滚动参数
5. 用户设置执行参数和访问凭证
6. 系统生成最终配置并提交执行

### 7.2 任务执行流程

1. 前端调用后端执行接口
2. 后端创建任务记录与执行记录
3. 爬虫引擎读取配置并开始采集
4. 结果写入本地文件与数据库记录
5. 前端通过状态更新查看执行进度
6. 用户查看结果或执行打包下载

### 7.3 模板复用流程

1. 用户从任务列表中选择已有任务
2. 将任务保存为模板
3. 模板中心统一管理模板
4. 后续使用模板快速恢复任务配置

## 8. 项目亮点

### 8.1 功能链路完整

本项目并不是单独实现某个页面或某个接口，而是完成了从用户登录、任务配置、任务执行到结果查看的完整闭环。

### 8.2 低代码思想落地

用户不需要从零手写完整爬虫代码，而是通过页面配置完成采集任务，这体现了低代码平台的核心思路。

### 8.3 模板化复用能力

将已有任务抽象为模板，减少重复配置，提高任务复用效率，这比单次任务执行更接近真实业务需求。

### 8.4 支持登录态采集

系统支持使用 Cookie 访问需要登录后才能查看的页面，这使平台适用于更多真实网页场景。

### 8.5 兼顾管理能力

除了基础采集功能，本项目还补充了任务组织、通知、工作台、后台管理等模块，使系统更接近一个完整平台，而不是单一工具。

---

## 9. 创新点总结

如果答辩老师问“你的创新点是什么”，可以从以下角度回答：

### 9.1 将网页采集流程抽象为低代码配置流程

把传统手写爬虫中的页面分析、字段选择、执行配置等流程进行可视化抽象，降低了使用门槛。

### 9.2 将任务沉淀为可复用模板

平台不仅能执行任务，还能把任务保存为模板，支持后续快速复用，这是对“配置资产化”的一种实现。

### 9.3 面向真实业务场景扩展登录态访问

系统设计了针对登录态页面的 Cookie 访问机制，使采集能力从公开页面延伸到部分需要身份验证的页面。

### 9.4 不只是抓取，还包括组织与运维管理

通过工作台、通知、后台管理等模块，将系统扩展为具有基础平台属性的应用，而不仅仅是单机爬虫工具。

说明：

本科毕设不一定要求“理论级创新”，更多是“设计思路有特点、实现路径完整、相较普通增删改查有应用价值”。  
所以这里的创新点更适合表述为“工程设计创新”和“应用实现创新”。

## 10. 技术选型说明

### 10.1 前端技术

- Vue 3
- TypeScript
- Vite
- Pinia
- Element Plus

选择原因：

- Vue 3 适合构建组件化前端界面
- TypeScript 提高可维护性
- Vite 构建速度快，适合开发调试
- Pinia 便于状态管理
- Element Plus 能够提升后台类系统开发效率

### 10.2 后端技术

- NestJS
- TypeORM
- MySQL
- Redis
- WebSocket

选择原因：

- NestJS 架构清晰，适合模块化后端开发
- TypeORM 方便实体建模和数据库操作
- MySQL 适合结构化业务数据存储
- Redis 适合验证码与会话类临时数据
- WebSocket 便于任务状态实时更新

### 10.3 爬虫技术

- Crawlee
- Playwright

选择原因：

- Crawlee 适合任务队列式爬取
- Playwright 能较好支持动态页面与浏览器自动化

主要实体可包括：

- 用户表
- 任务表
- 执行记录表
- 模板表
- 通知表
- 系统设置表
- Cookie 凭证表

设计思路：

- 用户和任务是一对多关系
- 任务和执行记录是一对多关系
- 用户和模板是一对多关系
- 用户和通知是一对多关系
- 部分敏感访问凭证单独存储，不直接暴露在普通任务配置中

## 13. 项目局限性

这部分答辩时一定要主动说，显得更客观。

### 13.1 对极复杂反爬网站支持有限

对于验证码、强风控、动态令牌校验等复杂网站，目前系统并不能完全通用支持。

### 13.2 通用性仍受页面结构影响

低代码配置方式适合规则较明确的网页，但对于结构极不稳定的网站，仍然需要人工调整配置。

### 13.3 目前主要面向本地实验部署

当前项目更适合本地实验与演示运行，在安全加固、部署规范、性能优化方面仍有进一步提升空间。

### 13.4 平台智能化程度还有提升空间

目前系统更偏“可视化配置”，未来还可以继续增强自动识别、自动推荐字段、自动纠错等能力。

## 16. 常见答辩问题与参考回答

### 问题 1：你为什么选择这个题目？

参考回答：

> 因为传统爬虫开发需要重复分析页面结构和编写采集逻辑，开发成本较高。我希望通过低代码方式把这类重复流程抽象出来，让用户通过配置完成任务，从而提高采集效率和复用性。

### 问题 2：你的项目“低代码”体现在哪里？

参考回答：

> 低代码主要体现在任务创建过程。用户不需要直接编写完整爬虫程序，而是通过页面预览、结构选择、字段映射、参数配置等方式完成任务定义，系统再根据这些配置驱动后端执行引擎完成采集。

### 问题 3：你的系统和普通爬虫脚本有什么区别？

参考回答：

> 普通爬虫脚本通常只解决一次性的抓取需求，而本系统更强调平台化，支持任务管理、模板复用、结果查看、通知和后台管理，更接近一个可持续使用的采集系统。

### 问题 4：项目的难点是什么？

参考回答：

> 我认为难点主要有三个：第一是前后端配置结构的一致性设计；第二是动态网页场景下的页面预览与字段提取；第三是任务执行、模板复用和结果管理之间的数据衔接。

### 问题 5：你的创新点是什么？

参考回答：

> 创新点主要体现在三个方面：第一是把网页采集流程做成低代码配置流程；第二是把任务沉淀为可复用模板；第三是在本科项目中进一步补充了工作台、通知和后台管理等平台能力。

### 问题 6：这个系统有哪些不足？

参考回答：

> 目前系统对复杂反爬网站支持仍有限，通用性也会受到网页结构差异影响。另外当前更偏本地实验运行，在生产级安全和部署方面还有进一步优化空间。

### 问题 7：如果继续完善，你会怎么做？

参考回答：

> 我会优先从三个方向继续优化：一是增强智能化识别能力，二是进一步提升页面元素拾取和配置的可视化体验，三是加强系统部署、日志监控和安全性。

### 问题 8：你在这个项目中承担了哪些工作？

参考回答：

> 我主要完成了系统整体方案设计、前后端功能实现、数据库结构设计、任务执行链路打通、模板与通知模块接入，以及界面调整和联调测试等工作。
